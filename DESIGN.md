## Problem Statement

AI is replacing traditional search as the primary way people discover restaurants. 45% of consumers now use AI tools (ChatGPT, Gemini, Perplexity) to find local services, up from 6% one year ago. 40% of Gen Z starts with AI chatbots for restaurant recommendations before opening Google.

But 83% of restaurants don't appear in AI-generated recommendations at all. AI recommends only 1.2% of local business locations. The restaurants that do get recommended average 3,424 Google reviews vs 955 for non-recommended ones. Star ratings above 4.4 barely matter... volume and structured data are the dominant signals.

Restaurant owners have no idea this is happening. They don't know AI search exists as a discovery channel, don't know they're invisible, and wouldn't know how to fix it if they did.

## Status Quo

Restaurants currently rely on:
- Google Business Profile (necessary but not sufficient for AI)
- Yelp, TripAdvisor, OpenTable listings (inconsistent, often outdated)
- Their own website (designed for humans, not AI agents)
- Word of mouth and Instagram

None of these are optimized for AI consumption. A typical restaurant website has beautiful photos, an emotional "Our Story" page, and a menu as a PDF or image. AI agents can't parse any of it. The restaurant is effectively invisible to the fastest-growing discovery channel.

## Target User & Narrowest Wedge

**Target user:** Independent restaurant owner or small restaurant group (2-10 locations) in a US metro area. Non-technical. Busy. Pays for results, not tools. Already paying $200-500/mo for Yelp ads or DoorDash commission and understands "pay for visibility."

**Narrowest wedge:** Auto-generate AI-optimized markdown pages for restaurants and host them on a high-authority domain. One product, one outcome: "show up when AI recommends restaurants."

## How It Works — Technical Mechanics

### 1. AI-Optimized Pages (Core Product)

Generate structured markdown pages for each restaurant, optimized for LLM consumption:

```markdown
# Joe's Pizza — San Francisco, Mission District

## Cuisine
Neapolitan pizza, Italian

## Price Range
$12-$22 per person

## Menu Highlights
- Margherita Pizza ($14) — San Marzano tomatoes, fresh mozzarella, basil
- Pepperoni ($16) — house-made pepperoni, hot honey drizzle

## Dietary Options
Vegetarian: yes | Vegan: 3 options | Gluten-free crust: available

## Atmosphere
Casual, lively, good for groups, outdoor patio

## Hours
Mon-Thu 11am-10pm | Fri-Sat 11am-11pm | Sun 12pm-9pm

## Location
456 Valencia St, San Francisco, CA 94103
2 blocks from 16th St BART, street parking available

## What People Say
"Best Neapolitan pizza outside of Italy" — 2,847 Google reviews, 4.6 stars
```

These pages are:
- Authored in markdown, **rendered to HTML** with embedded JSON-LD schema markup, served as standard web pages
- FAQ-format where possible (matches how users query AI)
- Rich in attributes: cuisine, price, vibe, dietary, location context, nearby landmarks
- Hosted on a single domain with proper crawler access (sitemap.xml, robots.txt allowing GPTBot/PerplexityBot/Google-Extended)

Data sources for auto-generation (in priority order):
1. **Google Places API (NOT Google Business Profile API)** — name, address, hours, cuisine, photos, reviews. Note: Places API provides up to 5 reviews, no menu data, no detailed hours for holidays. GBP API requires OAuth from the business owner, so it's only usable after the restaurant onboards and grants access.
2. **Restaurant's own website** — menu scraping via Playwright/Puppeteer. Handles HTML menus well. Text-based PDFs via OCR (Tesseract or cloud OCR). Image-heavy PDFs and image-only menus: use LLM vision (GPT-4o vision or Claude vision) to extract menu items. Manual transcription as last resort.
3. **DoorDash/UberEats/Grubhub listings** — backup menu source, often more structured than the restaurant's own site
4. **Review aggregation** — Google Places API for review snippets (limited to 5). For volume/sentiment analysis, scrape Google Maps reviews (TOS gray area, see Legal section).
5. **Manual input from restaurant owner** — final fallback for accuracy, and for attributes not available online (noise level, parking, private dining)

**Edge cases:**
- Restaurant with no website and no delivery listing: Google Business Profile + manual input only
- Menu changes frequently (specials, seasonal): owner can update via dashboard, or re-scrape on a schedule (weekly)
- Chain restaurants with identical menus: template once, customize per location (address, hours, reviews)

### 2. Schema Markup

Generate JSON-LD structured data for each restaurant:

```json
{
  "@context": "https://schema.org",
  "@type": "Restaurant",
  "name": "Joe's Pizza",
  "servesCuisine": ["Italian", "Pizza"],
  "priceRange": "$$",
  "address": {
    "@type": "PostalAddress",
    "streetAddress": "456 Valencia St",
    "addressLocality": "San Francisco",
    "addressRegion": "CA"
  },
  "aggregateRating": {
    "@type": "AggregateRating",
    "ratingValue": "4.6",
    "reviewCount": "2847"
  },
  "openingHoursSpecification": [...],
  "menu": "https://[platform-domain]/joes-pizza-sf/menu"
}
```

Schema makes the restaurant a parseable fact for AI models, not ambiguous text. Embedded in both the hosted page and injectable into the restaurant's own website.

### 3. Directory Consistency (Phase 2 — Not MVP)

Audit NAP (Name, Address, Phone) consistency across major directories:
- Google Business Profile (has API — read/write possible)
- Yelp (read-only — no public write API, manual updates required)
- TripAdvisor (read-only — no public write API)
- Apple Maps (via Apple Business Connect — limited API)
- Facebook (Graph API — read/write possible)
- OpenTable (partnership required)

**MVP scope:** Audit only. Generate a "consistency report" showing where data conflicts exist. Restaurant owner or platform team fixes manually. Automated sync is a Phase 2 feature and only feasible for Google and Facebook via APIs. Other platforms require manual updates or partnerships (similar to how Yext operates).

When LLMs see conflicting data across sources, confidence drops and the restaurant gets skipped. Consistency = trust signal = more AI mentions.

### 4. Query Intelligence

Probe AI models to discover high-intent restaurant queries by city/neighborhood:
- "best Thai food in [neighborhood]"
- "quiet restaurant for date night in [city]"
- "restaurants with outdoor seating near [landmark]"
- "cheap lunch spots in [area]"

**Method:** Automated prompting via OpenAI/Anthropic/Google APIs. For each city/neighborhood, run a batch of ~50 templated queries ("best [cuisine] in [area]", "restaurants near [landmark] for [occasion]") across ChatGPT, Gemini, and Perplexity. Log which restaurants appear and which attributes are mentioned. Run weekly as a cron job. Estimated API cost: ~$20-60/city/month (600 calls/month at $0.03-0.10 per call depending on model and response length).

**Phase 1 (MVP):** Manual query testing for SF only. Founder runs queries, logs results in a spreadsheet, uses patterns to inform page templates.
**Phase 2:** Automated query probing pipeline across multiple cities.

Reverse-engineer page content to answer these specific queries. This is the intelligence layer — knowing WHAT to write on the page, not just generating a template.

### 5. Measurement & Reporting

**The hard truth:** AI mention tracking is technically difficult. ChatGPT responses are non-deterministic and there is no official "what did you recommend?" API. This is an unsolved problem across the industry.

**Practical approach for MVP:**
- Run a fixed set of ~100 restaurant queries per city per week via AI APIs
- Log whether each client restaurant appears in the response
- Report: "Out of 100 relevant queries we tested, your restaurant appeared in X"
- This is sampling, not census. But it's measurable and repeatable.

**Tools that exist:** Local Falcon (early AI tracking), BrightLocal, Semrush (adding AI features). Can integrate these rather than building from scratch.

**What we can track reliably:**
- Page crawl status (is the AI-optimized page being indexed? Check via Google Search Console, Bing Webmaster)
- Referral traffic from AI platforms (ChatGPT sends traffic with identifiable referrer headers)
- Query appearance rate from our own sampling

**What we cannot track reliably (yet):**
- Total AI mentions across all user sessions (no API exists for this)
- Real-time mention alerts

Monthly report to restaurant owner: "We tested 100 AI queries relevant to your restaurant. You appeared in X responses, up from 0 last month. Here are the queries where you showed up."

## Legal & Compliance

- **Scraping risk:** Google Maps/Yelp scraping violates their TOS. Mitigation: use official APIs where available (Google Places API is legitimate), supplement with publicly displayed information only, and do not store raw scraped data long-term. If a platform sends a cease-and-desist, remove their data and fall back to restaurant-provided info.
- **Review data:** Only display aggregated review counts and sentiment, not full review text (reduces copyright/TOS risk).
- **Dietary claims liability:** Disclaimer on every page. AI-generated dietary info is labeled as "based on public menu data, confirm with restaurant." Owner review step catches obvious errors.
- **CCPA/privacy:** No consumer PII is collected. Restaurant business data (name, address, menu) is public information.

## Critical Technical Risks (Must Resolve Before Building)

### Domain Authority Cold Start (LOAD-BEARING)

The entire product depends on AI crawlers finding and trusting the hosted pages. A brand-new domain has zero authority. Options to investigate:

- **Option A: Buy an existing high-DA domain.** Expired domains with restaurant/food content and existing backlinks. Cost: $500-5,000. Risk: Google penalizes obvious domain repurposing.
- **Option B: Subdomain of an established site.** Partner with a food blog, restaurant directory, or media site. Their domain authority bootstraps yours. Risk: dependency on partner.
- **Option C: Build authority fast with volume.** Launch with 500+ restaurant pages at once (scrape and auto-generate for an entire city). Submit sitemap to Google, Bing, and AI crawlers. ChatGPT's browse feature and Perplexity actively crawl the web. GPTBot, Google-Extended, and PerplexityBot are the relevant crawler user-agents. Risk: still takes weeks/months.
- **Option D: Bypass the domain problem entirely.** Instead of hosting pages on your own domain, inject structured data (schema markup) directly into the restaurant's existing website, and submit the restaurant's own URL to AI crawlers. This removes the domain authority problem but requires access to the restaurant's site.

**Recommendation:** Lead with Option D (schema injection) in Phase 1. It sidesteps the cold-start problem entirely because the restaurant's existing domain already has authority. Run Option C (own domain with volume) in parallel as the long-term play. **Blocking milestone: "first crawl confirmed by GPTBot or PerplexityBot" must happen before onboarding paying customers.** Verify via server access logs checking for crawler user-agents (GPTBot, PerplexityBot, Google-Extended).

**Concrete plan for own domain:** Register a clean domain (e.g., pattern like "bestlocaleats.com" or "aidiningguide.com"). Deploy via Cloudflare Pages. Submit sitemap to Google Search Console, Bing Webmaster Tools. Ping AI crawlers via their documented submission URLs. Track crawl status in server logs daily.
